# ПитайБаце

„ПитайБаце“ е уеб приложение (тикет система), в което регистриран клиент
заплаща услуги за своя уебсайт и създава тикет, а експерт („пустиняк“)
обработва заявката до нейното решаване. Всяка поръчка може да породи точно
един тикет, комуникацията се води чрез съобщения, а чувствителните достъпи
до сайта и хостинга се съхраняват криптирани.

## Технологии

- Python 3.13, Django 5.2
- SQLite за разработка
- `cryptography` (Fernet) за криптиране на достъпите
- `python-dotenv` за конфигурация чрез `.env`
- Само ръчно писан CSS — без CSS framework-ци

## Структура на приложенията

- **accounts** — потребители (собствен `User` модел с роля `client` / `expert`),
  регистрация, вход/изход и клиентско табло.
- **departments** — отдели, към които се разпределят експертите и тикетите.
- **services** — каталог с услуги, които клиентите заплащат.
- **orders** — поръчки и (симулирано) плащане зад `PaymentGateway` интерфейс.
- **tickets** — тикети, съобщения, прикачени файлове, криптирани достъпи,
  workflow логиката (`services.py`) и management командата за неактивност.
- **logs** — `ActivityLog` (одит на действията).
- **staff** — експертен панел (само за експерти и администратори).

## Настройка за разработка

```bash
# 1. Виртуална среда
python -m venv venv
source venv/bin/activate

# 2. Зависимости
pip install -r requirements.txt

# 3. Конфигурация — копирайте примерния файл и попълнете стойностите
cp .env.example .env
#   SECRET_KEY  — таен ключ на Django
#   DEBUG       — True за разработка (тогава имейлите отиват в конзолата)
#   FERNET_KEY  — ключ за криптиране на достъпите, генерирайте с:
#     python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"

# 4. Миграции (създават базата и зареждат примерни отдели и услуги)
python manage.py migrate

# 5. Суперпотребител (администратор)
python manage.py createsuperuser

# 6. Стартиране
python manage.py runserver
```

### Примерни данни (fixtures)

Примерните **отдели** и **услуги** (напр. „Дезинфекция на сайто“, „Убав
хостинг за сайто“) се зареждат автоматично чрез data-миграция при
`python manage.py migrate` — не е нужна отделна команда за fixtures.
Суперпотребителят е обикновен Django `is_superuser` акаунт, без отделна роля;
клиенти се създават чрез регистрацията, а администратор може да превърне
клиент в експерт от експертния панел (секция „Потребители“).

## Автоматично връщане на неактивни тикети (cron)

Командата `check_inactive_tickets` намира тикетите в статус „Чакаме та“
(`waiting_reply`) или „Експерто бачка“ (`in_progress`), по които няма
активност повече от `TICKET_INACTIVITY_HOURS` часа (по подразбиране 48), и ги
връща в статус „отворен“. Клиентът получава имейл, а промяната се записва в
лога като системно действие (без конкретен извършител).

```bash
python manage.py check_inactive_tickets
```

Пример за ежечасно изпълнение чрез cron (`crontab -e`), с абсолютни пътища:

```cron
0 * * * * cd /path/to/pitaibace && /path/to/pitaibace/venv/bin/python manage.py check_inactive_tickets >> /var/log/pitaibace/inactive_tickets.log 2>&1
```

## Кратък преглед на архитектурата

- **Потребители и роли.** Собствен `User` модел с роля `client` или `expert`.
  Експертът задължително принадлежи към отдел. Администраторът е Django
  `is_superuser`.
- **Единствена точка за смяна на статус.** Всяка промяна на статуса на тикет
  минава през `tickets/services.py:change_status`, който валидира прехода,
  записва `ActivityLog` и изпраща имейл до клиента. Помощните действия
  (назначаване на експерт, ново съобщение, приключване, автоматичното връщане)
  също минават през него.
- **Контрол на достъпа.** Всички изгледи изискват вход. Клиентът вижда само
  своите тикети и поръчки; експертът — само тикетите на своя отдел; експертният
  панел е достъпен само за експерти и администратори (`expert_required`),
  а промяната на роли/отдели — само за администратори (`superuser_required`).
  Вътрешните бележки (`is_internal=True`) никога не се показват на клиента.
- **Криптирани достъпи.** Полетата за достъп в `Credential` се пазят криптирани
  с Fernet (`EncryptedTextField`) — в базата никога няма чист текст. Достъпите
  се показват в чист вид само на **назначения** експерт и само след изрично
  „Покажи достъпите“, като всяко разкриване се записва в `ActivityLog`.
- **Плащане.** Плащането е симулирано зад `PaymentGateway` интерфейс, така че
  реални интеграции (карта / PayPal / ePay) могат да се добавят по-късно.
- **Имейли.** Използва се стандартният Django email backend — конзолен при
  `DEBUG=True`, SMTP (от `.env`) в останалите случаи.
- **Качване на файлове.** Прикачените файлове се валидират по размер
  (макс. 5 MB) и разширение.
